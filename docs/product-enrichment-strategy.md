# Product Description and Image Enrichment Strategy

Greenway’s POS export should remain the source of truth for inventory, pricing, package size, potency, category, brand, strain name, and strain type. Product descriptions and product images need a separate enrichment layer because Cultivera does not reliably export those assets. The transformer should join this enrichment layer onto the POS feed every time the menu is generated.

The best long-term structure is a small website-managed product enrichment registry. Each enrichment record should be keyed by a stable product identity, not by inventory ID alone, because inventory IDs and barcodes can change when the same sellable item is received again. The recommended key is a normalized combination of brand, original POS product name, package size, and category, with optional fallback matching by brand plus strain plus category plus package size. The transformer already preserves both the customer-facing `name` and the original `productName`; `productName` should be treated as the canonical POS product-name identity for joining descriptions and images.

A practical first version can be a CSV or JSON file committed with the website repository, for example `data/product-enrichment.csv` or `data/product-enrichment.json`. Suggested columns are `brand`, `productName`, `category`, `packageKey`, `strainName`, `description`, `imageUrl`, `imageAlt`, `imageStatus`, `lastReviewedAt`, and `notes`. The transformer should load this registry, build normalized lookup keys, and apply description/image fields after matching Products + Inventories. If a product has a description in the uploaded Products workbook, that value should win for now. If the Products workbook description is blank, the enrichment registry should fill it. If both are blank, the transformer should output `Description pending for {name}` and add that product to a generated review report.

For new products, the workflow should be: upload/export the latest Products and Inventories spreadsheets, run the transformer, and let it produce a `missing_enrichment_report.csv` listing products that are live or in stock but lack a real description or image. That report becomes the daily/weekly content worklist. The team can add descriptions and image URLs to the enrichment registry without touching website code. The next transform automatically joins those enrichments into the menu.

For images, use stable externally hosted image URLs at first. The cleanest option is a dedicated folder in an asset manager such as Cloudinary, Vercel Blob, S3, or a GitHub-backed public assets folder. Each image record should reference a URL and an alt text string. Avoid storing image binary files directly inside the product spreadsheets. The transformer should not guess images; it should either use an exact enrichment match or leave the image in the current generated/mock visual state and flag it in the missing enrichment report.

To keep data clean, the transformer should generate audit files every run: matched products, unmatched inventory rows, ambiguous product matches, missing descriptions, missing images, duplicate enrichment keys, and stale enrichment records that no longer appear in the current POS feed. This makes the process safe: old descriptions/images can stay in the registry for products that may return later, while stale records are visible and can be archived if desired.

The recommended roadmap is to start with the current `Description` column in `Products_with_Descriptions_and_Strain_Type.xlsx`, then add a persistent enrichment registry before the next product refresh. Later, this can be upgraded into a lightweight admin page or Airtable/Google Sheet/CMS-backed workflow, but the same rule should remain: POS controls operational data, enrichment registry controls marketing content, and the transformer joins them with transparent review reports.

<h1>Greenway Marijuana Back Office Strategy Report</h1><p>Prepared for the next major project phase: building a professional employee/admin back office for content, media, products, vendors, POS imports, orders, promotions, and reporting.</p><h2>TL;DR</h2><p>The Greenway website is currently a polished Next.js 16 / React 19 / TypeScript / Tailwind application with a strong customer-facing menu, checkout-style pickup flow, SEO foundation, POS transformer, blog content model, vendor directory, daily specials logic, and static brand/media assets. What it does <strong>not</strong> yet have is a true backend database, authenticated staff portal, persistent order system, CMS, media library, or admin-controlled publishing workflow.</p><p>The right next phase is to build a real back office around the systems already present. The website should keep its beautiful customer front end, but the source of truth should move from committed TypeScript/JSON/static files into a database-backed admin system. Employees should be able to upload Cultivera POS spreadsheets, review transformer reports, approve menu updates, manage graphics and banners, write blog posts, edit approved site text, manage vendor/brand profiles, choose Thursday sale brands, review incoming pickup orders, and view operational reports. The current POS transformer should not be thrown away; it should become an internal import job that runs after spreadsheet upload, produces diagnostics, stages menu changes for review, and publishes only after staff approval.</p><p>Recommended architecture: keep Next.js/Vercel for the website and admin UI, add Postgres as the core database, add Prisma or Drizzle for schema/migrations, add Auth.js/Clerk/Supabase Auth for secure staff login with roles, add object storage for images/files, and add background job processing for POS imports, image processing, reports, and future bot automation. Supabase is likely the fastest all-in-one choice because it provides Postgres, auth, storage, row-level security, and edge-friendly APIs, while still allowing a professional custom admin UI inside the existing app.</p><p>Build this in phases. Phase 1 should be foundation: database, auth, roles, audit log, media library, and admin shell. Phase 2 should be POS upload/import/staged publish because that becomes the backbone of menu accuracy. Phase 3 should be CMS controls: banners, blog posts, site text, page metadata, vendors/brands/products/assets. Phase 4 should be order management: real server-side orders, statuses, staff dashboard, notifications, printable receipts, and pickup workflow. Phase 5 should be reporting: transformer diagnostics, inventory health, product/category/brand metrics, sales/order stats, promotion performance, and export tools. Phase 6 can add future automation: POS-fetch bot, scheduled imports, AI-assisted product copy, vendor enrichment, and more advanced analytics.</p><p>The main strategic principle: do <strong>not</strong> make employees edit code. The website is now professional enough that the back office must be equally professional: role-based, auditable, safe to use, preview-before-publish, compliant, and built around clear workflows.</p><hr><h2>1. What exists today, based on the current codebase</h2><p>The current repository is a lean, high-quality customer website rather than a full backend application. The stack is Next.js 16 App Router, React 19, TypeScript, Tailwind CSS v4, and Vercel deployment. The package scripts show that the production build runs <code>npm run transform:pos &amp;&amp; next build</code>, which means the POS transformer is already part of the build pipeline.</p><p>The site has routes for home, menu, product detail pages, specials, about, locations, loyalty, blog, FAQ, price match, vendor delivery, legal/privacy pages, checkout, checkout confirmation, sitemap, robots, and an internal loyalty signup review route. The site has a strong SEO helper system in <code>src/lib/seo/seo.ts</code> and JSON-LD rendering via <code>src/components/seo/JsonLd.tsx</code>. This should be preserved and eventually made admin-configurable where appropriate.</p><p>The menu data currently comes from a POS transformer. Raw Cultivera spreadsheets are expected locally at <code>pos-data/raw/PRODUCTS.xlsx</code> and <code>pos-data/raw/INVENTORIES.xlsx</code>. The transformer writes <code>src/data/pos-menu-preview.json</code>, <code>src/data/pos-menu-sample-preview.json</code>, and <code>src/data/vendors.json</code>. The production build can fall back to committed JSON if the raw workbooks are not present. That is good for current deployment stability, but a back office should replace this manual developer workflow with an authenticated upload, transform, review, and publish process.</p><p>The product model already has useful fields: product id, name, productName, brand, vendor, category, filter categories, POS inventory type/category, strain type, strain name, THC/CBD and compounds, description, price, inventory status, hidden flag/reason, and variants with label, price, inventory level, and medical flag. That gives us a strong starting shape for database tables.</p><p>Checkout is currently client-side. Orders are generated in the browser, stored in <code>sessionStorage</code>, and shown on the confirmation page. Cart state and runtime inventory decrement use <code>localStorage</code>. This gives customers a good flow visually, but it is not yet a real operational order management system. A real back office should move orders to the server and database, give staff a dashboard, and trigger notifications.</p><p>Loyalty signups are closer to server-side: the API validates form submissions, optionally emails through Resend, and appends JSON lines to <code>storage/loyalty-signups.jsonl</code>. The current admin page reads that file and explicitly warns that it is a preview page and must be protected by authentication before public use. The back office should replace JSONL storage with a database table and make loyalty submissions a proper staff queue.</p><p>Blog content currently lives in TypeScript objects under <code>src/lib/blog/posts.ts</code>, with an employee update guide comment explaining how to copy objects and add images to <code>public/blog/</code>. That is a strong signal that the blog is ready for a CMS. Employees should not edit TypeScript; they need a friendly editor with drafts, previews, publish dates, categories, images, SEO fields, and newsletter/PDF support.</p><p>Vendor data is generated from POS counts in <code>src/data/vendors.json</code>. The vendor directory currently displays 112 vendors with placeholder logos and placeholder descriptions. The code comments say the logos and partner descriptions are placeholders pending final vendor assets. This maps directly to your desired vendor/brand database: logos, mission statements, about info, product pictures, descriptions, and contact details.</p><p>Specials logic is currently hardcoded in <code>src/lib/specials/daily-deals.ts</code>. Monday/Tuesday/Wednesday/Friday/Saturday/Sunday deals are category/storewide driven. Thursday Top Shelf is brand-based, with a hardcoded <code>topShelfThursdayBrands</code> array. The back office should turn this into a promotions module where staff can choose brands, categories, schedules, discount rules, and publish windows.</p><p>Static graphics and business info currently live in <code>public/</code> and <code>src/content/business.ts</code>. The back office should create a media library and content settings layer so banners, hero images, logo references, page copy, and business info can be changed without code changes, while keeping approval controls and audit history.</p><hr><h2>2. Core goal of the back office</h2><p>The goal should be to create a single, professional internal operating system for the website. It should serve four audiences.</p><p>First, store employees need a simple day-to-day interface. They should be able to upload POS files, review warnings, update banners, choose sale brands, create blog/newsletter posts, manage incoming pickup orders, and edit controlled text without technical knowledge.</p><p>Second, managers and owners need control, visibility, and safety. They need roles, approvals, audit logs, drafts, scheduled publishing, reports, exports, and confidence that no one accidentally breaks the live site.</p><p>Third, customers need a more accurate and richer front end. The front end should draw from the database: real product photos, brand logos, vendor profiles, richer descriptions, current promotions, accurate order status, and polished content.</p><p>Fourth, future automation needs clean foundations. A POS-fetch bot, auto-importer, vendor enrichment bot, image processing, sales analytics, and AI-assisted content tools are all easier if the database and back-office workflows are designed properly now.</p><p>The back office should not be an afterthought. It should match the quality of the customer site: same visual identity, same care in UX, same compliance awareness, same SEO discipline, and same attention to maintainable architecture.</p><hr><h2>3. Recommended technical architecture</h2><h3>3.1 Keep the current front end, add a real backend foundation</h3><p>The current Next.js app should remain the main application. It already has routes, styling, SEO, menu UI, checkout UI, and product pages. The new back office can live under protected routes such as <code>/admin</code>, <code>/admin/menu</code>, <code>/admin/orders</code>, <code>/admin/content</code>, and <code>/admin/reports</code>.</p><p>The biggest architectural change is adding persistent backend services:</p><ul> <li><strong>Database:</strong> Postgres.</li> <li><strong>ORM/schema layer:</strong> Prisma or Drizzle. Prisma is friendlier and fast to develop; Drizzle is lighter and very TypeScript-native. Either works.</li> <li><strong>Auth:</strong> Supabase Auth, Clerk, or Auth.js. Because this project also needs storage and Postgres, Supabase is probably the most efficient all-in-one option.</li> <li><strong>File/object storage:</strong> Supabase Storage, S3, Cloudflare R2, or Vercel Blob for images, spreadsheets, PDFs, newsletters, and generated reports.</li> <li><strong>Background jobs:</strong> Inngest, Trigger.dev, QStash, or a small internal job table/worker for POS imports and report generation.</li> <li><strong>Email/notifications:</strong> Resend is already compatible with the existing loyalty flow; keep it for order alerts, loyalty alerts, import summaries, and staff notifications.</li> <li><strong>Charts:</strong> Recharts, Tremor, Nivo, or lightweight SVG charts for admin dashboards.</li> </ul><h3>3.2 Why Supabase is likely the best fit</h3><p>Supabase would give you Postgres, authentication, storage, row-level security, and admin APIs in one package. For this project, that matters because you need database records, file uploads, private staff access, media storage, spreadsheet storage, and future automation. It also avoids overbuilding infrastructure too early.</p><p>A more custom stack with Neon Postgres + Clerk + S3/R2 is also excellent, but it has more moving parts. If speed and maintainability matter, Supabase plus a custom Next.js admin UI is the most practical route.</p><h3>3.3 Important rule: database content should not destroy performance</h3><p>The current site benefits from static generation and fast Vercel delivery. Moving everything to a database does not mean every page should become slow and dynamic. The ideal model is:</p><ul> <li>Store editable content in the database.</li> <li>Use caching and revalidation for customer pages.</li> <li>Publish approved menu/content snapshots that the front end can read quickly.</li> <li>Keep product pages cacheable where possible.</li> <li>Trigger revalidation after publish events.</li> </ul><p>For menu data, the safest approach is a versioned publish model: employees import POS data into a staging version, review diagnostics, approve it, then publish. The front end reads the currently published version.</p><hr><h2>4. Major back-office modules</h2><h3>4.1 Admin dashboard home</h3><p>The admin home should be a high-level command center. It should show today’s order count, pending pickup orders, current active sale, last POS import time, import warnings, inventory health, unpublished drafts, pending loyalty signups, and quick actions.</p><p>Recommended dashboard cards:</p><ul> <li>Orders awaiting review / in progress / ready / completed.</li> <li>Current live menu version and last import timestamp.</li> <li>POS import health: errors, warnings, hidden products, unmatched rows.</li> <li>Current active special and upcoming scheduled specials.</li> <li>Blog drafts awaiting publish.</li> <li>Recently uploaded media.</li> <li>Loyalty signups awaiting manual POS entry.</li> <li>Low-stock and out-of-stock product counts.</li> <li>Top brands/categories by current inventory depth.</li> </ul><h3>4.2 User accounts, roles, and permissions</h3><p>This needs to be built early. Every write action should have a staff identity attached. At minimum:</p><ul> <li><strong>Owner/Admin:</strong> full access, user management, settings, publish approvals.</li> <li><strong>Manager:</strong> manage menu imports, promotions, orders, content, vendors, reports.</li> <li><strong>Content Editor:</strong> blog, banners, page text, media library; no order/customer export access unless granted.</li> <li><strong>Budtender/Staff:</strong> order dashboard, loyalty review, limited reporting.</li> <li><strong>Read-only/Analyst:</strong> reporting and exports only.</li> </ul><p>Every important action should create an audit log entry: who changed what, when, old value, new value, and whether it was published.</p><h3>4.3 Media and graphics library</h3><p>This is one of your main requests and should be a flagship module. The current site uses static files in <code>public/brand</code>, <code>public/home</code>, <code>public/vendors</code>, <code>public/blog</code>, and <code>public/accessories</code>. The back office should turn this into a searchable media library.</p><p>Features:</p><ul> <li>Upload images, SVGs, PDFs, and newsletters.</li> <li>Tag media by usage: hero banner, home banner, product image, brand logo, vendor logo, blog image, newsletter, compliance asset, storefront photo, accessory image.</li> <li>Store alt text, title, description, photographer/source, license/approval status, and focal point/crop settings.</li> <li>Generate thumbnails and optimized responsive sizes.</li> <li>Replace images in controlled slots: home hero, brand banner, category banner, loyalty hero, vendor hero, vendor section banner, blog images, product cards, product detail pages, brand/vendor logos.</li> <li>Track where each asset is used before deletion.</li> <li>Support draft replacement: upload a new banner and preview it before publishing.</li> </ul><p>This should not just be a file dump. It should be a curated asset manager with metadata, usage tracking, and safe publishing.</p><h3>4.4 Site content editor</h3><p>The current site text is spread across components and content files. A back office should allow editing approved text areas without changing code. This should be structured, not a chaotic page builder.</p><p>Recommended model: define editable content blocks by key. Example keys:</p><ul> <li><code>home.hero.title</code></li> <li><code>home.hero.subtitle</code></li> <li><code>menu.hero.title</code></li> <li><code>menu.hero.subtitle</code></li> <li><code>loyalty.hero.title</code></li> <li><code>vendors.outreach.heading</code></li> <li><code>footer.compliance.warning</code></li> <li><code>business.hours.display</code></li> <li><code>specials.hero.subtitle</code></li> </ul><p>For each content block, store:</p><ul> <li>page/section/key</li> <li>current published value</li> <li>draft value</li> <li>field type: plain text, rich text, markdown, URL, phone, email, image reference</li> <li>validation rules</li> <li>SEO impact flag</li> <li>last edited by / last published by</li> </ul><p>Do not start with a fully free-form drag-and-drop page builder. That would risk damaging the polished design. Start with controlled fields that preserve the design system.</p><h3>4.5 Blog and newsletter CMS</h3><p>The existing blog object model is ready to become database-backed. The admin should support:</p><ul> <li>Create/edit/delete posts.</li> <li>Draft, scheduled, published, archived states.</li> <li>Categories matching current model: Products, Deals, Culture, Newsletter; allow future categories if desired.</li> <li>Slug editor with uniqueness validation.</li> <li>Excerpt, author, publish date, hero image, alt text.</li> <li>Rich text editor or markdown editor with preview.</li> <li>SEO title, SEO description, canonical, OG image.</li> <li>Newsletter mode: upload PDF, upload/generated page images, set first page as preview image.</li> <li>Preview route before publish.</li> <li>Auto-generate sitemap entries for published posts.</li> </ul><p>Because cannabis content has compliance implications, blog publishing should support an optional approval step.</p><h3>4.6 Vendor and brand database</h3><p>This is central to your vision. The current vendor directory is generated from POS vendor names and product counts. The new system should separate vendor/business identity from POS strings while preserving mapping.</p><p>Recommended features:</p><ul> <li>Vendor/brand profile records.</li> <li>Official display name plus POS aliases.</li> <li>Logo, brand mark, hero image, product images.</li> <li>Mission statement, about copy, product philosophy, growing/processing style.</li> <li>Contact person, email, phone, website, Instagram/social links.</li> <li>License info if needed.</li> <li>Notes for internal buyers.</li> <li>Vendor day scheduling notes.</li> <li>Product count and active menu products derived from menu data.</li> <li>Public/private fields: some info visible on the customer site, some internal only.</li> <li>Mapping tools to merge duplicate POS vendor names such as inconsistent capitalization or legal names.</li> </ul><p>The front end can then use this database to show real vendor cards, brand logos on product cards, richer product pages, brand landing pages, and vendor filter pages.</p><h3>4.7 Product enrichment database</h3><p>The POS data should remain the operational source of truth for price, inventory, and raw product identity. But POS exports usually do not contain beautiful product imagery, polished descriptions, brand storytelling, SEO text, or staff recommendations. The back office should allow enrichment layered on top of POS data.</p><p>Recommended product enrichment fields:</p><ul> <li>Canonical product id / POS product key.</li> <li>Display name override.</li> <li>Description override.</li> <li>Product image(s).</li> <li>Brand logo relationship.</li> <li>Vendor relationship.</li> <li>Strain notes / terpene notes where approved.</li> <li>Staff pick flag.</li> <li>Featured product flag.</li> <li>Hide from website flag with reason.</li> <li>Compliance notes/internal notes.</li> <li>SEO title/description override.</li> <li>Product tags: new arrival, best seller, staff pick, local favorite, high CBD, etc.</li> </ul><p>Important: preserve the distinction between POS truth and marketing enrichment. Staff should not accidentally edit inventory or price in the enrichment layer unless that is explicitly allowed.</p><h3>4.8 POS spreadsheet upload and import workflow</h3><p>This should be one of the most important modules. Today the developer workflow is: place Cultivera files in <code>pos-data/raw</code>, run transformer, commit generated JSON. The admin workflow should become:</p><ol> <li>Employee uploads <code>PRODUCTS.xlsx</code> and <code>INVENTORIES.xlsx</code>.</li> <li>System stores the raw files in private storage with timestamp, uploader, file hash, and original filename.</li> <li>Background job runs the existing transformer logic against those files.</li> <li>Transformer writes structured import results to database tables rather than only JSON files.</li> <li>System displays an import review screen with diagnostics.</li> <li>Staff reviews counts, warnings, hidden products, unmatched rows, category mapping issues, price changes, inventory changes, new products, removed products, and vendor changes.</li> <li>If serious errors exist, publishing is blocked.</li> <li>If warnings are acceptable, manager approves and publishes the new menu version.</li> <li>Front end revalidates menu/product pages and starts reading the new published version.</li> <li>Import files, diagnostics, and approval decision remain permanently auditable.</li> </ol><p>The current transformer is valuable and should be refactored into reusable library functions rather than only a CLI script. The CLI can remain for local developer use, but the admin upload should call the same core transform functions.</p><p>Recommended import screens:</p><ul> <li>Upload files.</li> <li>Import history.</li> <li>Import detail summary.</li> <li>Error/warning diagnostics table.</li> <li>Hidden items review.</li> <li>New/changed/removed products diff.</li> <li>Category mapping review.</li> <li>Vendor/brand alias suggestions.</li> <li>Publish approval screen.</li> <li>Download reports: transform summary, anomaly report, hidden-items spreadsheet.</li> </ul><h3>4.9 Promotions and specials manager</h3><p>The current specials system is hardcoded in <code>daily-deals.ts</code>, including Thursday brands. The back office should make promotions editable while preserving safe rules.</p><p>Features:</p><ul> <li>Daily recurring deal rules: Monday Munchie, Tuesday Doobie, Wednesday Wax, Thursday Top Shelf, Friday Ounce, Saturday/Sunday storewide.</li> <li>Brand-based promotions, especially Thursday.</li> <li>Category-based promotions.</li> <li>Product-specific promotions.</li> <li>Clearance/50% off collection management.</li> <li>Start/end dates and time windows.</li> <li>Draft/scheduled/active/expired statuses.</li> <li>Discount types: percentage, fixed amount, buy-X-get-Y, threshold/spend bonus, multi-item tier.</li> <li>Compliance disclaimer fields.</li> <li>Preview affected products before publishing.</li> <li>Conflict detection: one product qualifying for multiple specials.</li> <li>Store-timezone correctness: America/Los_Angeles.</li> </ul><p>For Thursday specifically, staff should see a list of brands from the current menu, select the 4–5 participating brands, preview the products affected, add internal notes, and schedule/publish the selection.</p><h3>4.10 Order management</h3><p>The current checkout flow looks real but orders are not persisted server-side. A back office should make pickup orders operational.</p><p>Core order features:</p><ul> <li>Server-side order creation API.</li> <li>Database-backed order number generation.</li> <li>Customer info: first name, last name, phone, email, birthday/age confirmation.</li> <li>Order lines with product, variant, price, tax breakdown, discount, quantity, source menu version.</li> <li>Status workflow: new, acknowledged, preparing, ready for pickup, completed, canceled, no-show.</li> <li>Staff notes and customer notes.</li> <li>Pickup window if desired.</li> <li>Order activity timeline.</li> <li>Print-friendly receipt/pick ticket.</li> <li>Email notification to customer and staff.</li> <li>Optional SMS later.</li> <li>Inventory reservation behavior: soft reserve until pickup/cancel/expiration.</li> </ul><p>The admin order screen should be extremely simple for staff during busy store hours. It needs large readable cards, clear status buttons, search by name/phone/order number, filters by status, and a print button.</p><p>Important compliance/operational note: no online payment should be collected unless Greenway later implements a compliant payment workflow. The order should remain a pickup reservation / order request, with final pricing/tax/purchase limits confirmed in store.</p><h3>4.11 Loyalty signup management</h3><p>Loyalty signups should move from JSONL file storage to a database table. The admin should show:</p><ul> <li>New signups.</li> <li>Export CSV.</li> <li>Mark as entered into POS.</li> <li>Duplicate detection by email/phone.</li> <li>Consent record with timestamp/signature.</li> <li>Notification status.</li> <li>Staff notes.</li> <li>Search/filter.</li> </ul><p>This could later integrate directly with whatever POS/CRM loyalty system is available.</p><h3>4.12 Reports and analytics</h3><p>The reports section should combine transformer diagnostics, menu/inventory analytics, order analytics, and content/performance reporting.</p><p>Initial reports grounded in the current transformer:</p><ul> <li>Import summary: total rows, matched products, unmatched rows, generated menu items, variants, hidden items, vendors, warnings, info messages.</li> <li>Hidden items report: hidden reason, POS product, inventory row, category, vendor, price, stock.</li> <li>Category mapping report: POS categories mapped to website categories, unknown categories, counts.</li> <li>Vendor report: vendor/product counts, duplicate/alias candidates, missing logos/descriptions.</li> <li>Inventory health report: out-of-stock, low-stock, high-stock, products with zero price, products with suspicious potency, products without descriptions.</li> <li>Price-change report: products whose price changed since previous import.</li> <li>New/removed products report.</li> <li>Product enrichment gaps: missing image, missing brand logo, missing vendor profile, placeholder description.</li> </ul><p>Order/sales-style reports once real orders exist:</p><ul> <li>Orders by day/week/month.</li> <li>Revenue/reservation totals using out-the-door pricing.</li> <li>Average order value.</li> <li>Items per order.</li> <li>Top products by ordered quantity.</li> <li>Top brands/categories.</li> <li>Discount/promotion usage.</li> <li>Canceled/no-show rate.</li> <li>Pickup completion time.</li> </ul><p>Content/media reports:</p><ul> <li>Blog posts by status.</li> <li>Pages missing SEO fields.</li> <li>Media assets missing alt text.</li> <li>Unused assets.</li> <li>Vendor profiles incomplete.</li> </ul><p>The reports module should include charts, tables, CSV export, date filters, and saved report presets. Early charts can be simple but polished: line charts for orders, bar charts for top categories/brands, donut charts for inventory by category, stacked bars for import warnings by severity.</p><h3>4.13 SEO and publishing controls</h3><p>The recent SEO work gives the site an excellent technical foundation. The admin should expose safe SEO fields:</p><ul> <li>Page SEO title and description.</li> <li>Canonical path if needed.</li> <li>OG image selection.</li> <li>Product SEO override.</li> <li>Blog SEO fields.</li> <li>Vendor/brand page SEO fields once those pages exist.</li> <li>Sitemap inclusion/exclusion controls for published content.</li> <li>Noindex toggles for internal/temporary pages.</li> </ul><p>The system should validate title/description lengths and show a Google-style preview.</p><h3>4.14 Compliance and safety controls</h3><p>Cannabis websites need guardrails. ## 11. ADDENDUM — Additional findings from deep data + codebase analysis (SuperNinja)

This addendum supplements the original report with concrete findings discovered while reading every key file in the repository, analyzing the two Cultivera workbooks row-by-row, and inspecting the live site. It also records gaps the original report did not yet cover.

### 11.1 Hard data facts the back office must design around
The Products workbook (`PRODUCTS.xlsx`) contains **3,311 product rows** across 20 columns and is the master metadata source: `Product Name`, `Inventory Type`, `Category`, `Brand`, `Type` (strain type), `Strain`, `UOM`, `Package Size`, price model fields, `Price`, vendor price fields, `Cannabis Y/N`, `Pre Packed/Bulk`, `Tax Exempt Y/N`, `Vendor Only Y/N`, `Image Attached Y/N`, and `Description`. The Inventories workbook (`INVENTORIES.xlsx`) contains **3,917 rows** across 32 columns and is the live operational source: `Id`, `Location`, `Barcode`, `Alias`, `Product`, `Category`, `InventoryType`, `Strain`, `Brand`, `Vendor`, `Product Price`, `Cost`, `Units Available For Sale`, `Units In Stock`, `Package Size`, `Storage Location`, `Is Medical`, `Is Cannabis`, `Is Sample`, `Lab`, `[COA Y/N]`, `Cbd`, `Cbda`, `Thc`, `Thca`, `Total`, `Terpene Total`, `Units On Hold`, `Quantity Sold`, `Quantity Purchased`, `Expiration date`, and `Received date`.

The single most important operational discovery is that **`Image Attached Y/N` is `N` for all 3,311 products** — there is not one product image in the POS export. This makes the media library and product-image workflow the highest-value enrichment effort, not a nice-to-have. By contrast, **every product already has a written description**, so description enrichment is mostly about polishing/standardizing and filling strain-level gaps rather than writing from scratch.

The data also reveals taxonomy that the schema should respect: **172 brands in Products / 183 in Inventories, 121 raw vendor strings (which collapse to ~105 canonical vendor groups after de-duplication), 46 POS product categories that map down to 21 website categories, and 1,707 unique strain names (1,417 with a resolvable indica/sativa/hybrid type).** The vendor and brand alias-merge tooling is therefore not optional polish — the raw POS strings genuinely contain duplicates and casing variants that must be reconciled to one canonical profile.

### 11.2 Vendor ↔ brand relationship is real and must be modeled as such
Inventories carries both `Brand` and `Vendor`, and they are distinct: many vendors carry multiple brands, and a brand maps to a vendor. The original report correctly hedged on whether to combine them; the data confirms they should stay separate, with `brands.vendorId` linking up to a `vendors` record, plus alias tables on both. The folder database built alongside this report already encodes this exact hierarchy (`vendor → brands → products`), so the database seed can be driven directly from that structure.

### 11.3 The organized folder database (new deliverable)
A new top-level `GREENWAY WEBSITE/` operating folder has been created as the human-and-software-readable source of truth. It contains: a `database/vendors/` tree with one folder per vendor, each holding an editable `vendor.json`, a `logo/` drop folder, and a `brands/` tree where every brand has `brand.json`, a `logo/` folder, and a `products/` tree where every product has `product.json` (POS metadata + empty enrichment fields) and an `images/` drop folder; a `database/strains/STRAINS_MASTER.xlsx` reference listing all 1,707 strains with type, the vendor-provided description, and blank "trusted description / source" columns to fill gaps; a `database/categories/` manifest of website categories and POS category usage; `intake/` folders for dropping fresh POS exports and loose media; a `transformer/` workspace mirroring the existing pipeline's inputs/outputs/diagnostics; a curated `media-library/` sorted by usage; `content/` for editable site text, legal, blog, and FAQ; and `exports/` for generated downloads. This gives the back office predictable read/write paths and gives staff a place to find or drop anything without touching code.

### 11.4 Gaps the original report did not explicitly call out
Several practical needs surfaced during analysis that should be added to scope. First, an **inventory intake/archive convention**: every uploaded POS export should be retained with a timestamp and file hash so imports are fully auditable and reproducible — the `intake/pos-uploads/archive/` folder formalizes this. Second, a **media metadata sidecar convention** (a small JSON next to each asset capturing alt text, source, license, focal point, and usage) so the eventual `media_assets` table can be seeded from the folder library rather than re-entered. Third, the existing client-side cart applies sale math in two places (`daily-deals.ts` card preview vs the cart engine as the authoritative charge); the promotions module must preserve that "card shows headline best-case price, cart computes exact charge" contract so the migration does not change displayed prices. Fourth, **vendor de-duplication is a first-class task** with measurable value (121 → ~105), not cleanup. Fifth, a **strain reference table** is needed precisely because, although products have descriptions, strain-level effect/terpene/lineage data is inconsistent and should be backfilled from trusted sources with citations. Sixth, the admin UI must be **mobile-first for budtenders** working the counter on phones/tablets — not merely responsive as an afterthought.

### 11.5 Confirmed design system for the admin UI
The live site and `globals.css` confirm the exact tokens the back office must reuse so it feels like one product: background `#000000`, foreground `#ffffff`, Greenway green `#7ed957`, dark green `#12351f`, gold `#ffd700`, orange `#ff7f00`, charcoal `#1a1a1a`, with the flowing script wordmark in headers and the orange primary-action button style seen on the age gate. The admin should adopt these tokens directly rather than introducing a new palette.

### 11.6 Recommended first build slice
Given the above, the first development slice should remain the secure foundation (auth, `/admin` shell, roles, audit log, media storage, dashboard skeleton), because every later module depends on it. The folder database delivered with this report makes the subsequent vendor/brand/product and media slices fast to seed. A complete, handoff-ready slice plan lives in `GREENWAY WEBSITE/docs/BACK_OFFICE_BUILD_TODO.md`.

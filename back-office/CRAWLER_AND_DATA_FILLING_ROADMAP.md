# Work-Horse Crawler + Data-Filling Roadmap

> **Owner's headline goal (verbatim intent):** Build the web crawler ("work-horse") so we can
> *fill in the missing gaps* — **vendors and all their info, brands and brand info, and product
> images**. Images may be (a) exact product matches, (b) brand/vendor-specific shots that clearly
> show the product even without the name on them (a branded joint tube / flower jar / boxed
> edible/drink shown without a title), (c) a *fair, honest substitute* when nothing else can be
> found, or (d) an **approved image-substitute KB** entry chosen by product-type / category so a
> product card is never blank.
>
> Two prerequisites the owner asked for **before** crawl4ai:
> 1. **Expand the bottom of the Knowledge Base page** to allow **manual entry of a new strain** with
>    *all* the fields the KB strain table uses/needs (NOT brand — brand info lives on the Vendors
>    page and will be filled by the AI later). So staff can add verified strains they find or are
>    told about by suppliers, growing the validated list without a deploy.
> 2. Stand up the **approved image-substitute KB** for every product type & category across
>    INVENTORIES + PRODUCTS, so blank cards always get a sensible, honest fallback image.
>
> **Standing rules respected throughout:**
> - One **PR per slice** (squash-merge, delete branch, sync main). Updates posted in chat as we go.
> - **AI/crawler output is DRAFTS ONLY** — an employee validates before anything publishes.
> - **Supabase** is the backend (locked). Migrations are **idempotent** and **applied MANUALLY** by
>   the owner in the Supabase SQL editor.
> - **POS is the source of truth for price & inventory** — crawler/AI never touch them.
> - **WA I-502 advertising compliance** is non-negotiable (no medical/effect claims, no minor appeal,
>   honest substitutes only — never misrepresent a product).
> - Everything keyed by the **stable POS product key** so enrichment survives menu re-imports.
> - This roadmap **builds on** `AI_ENHANCEMENT_ROADMAP.md` (AI-1..AI-3 done; AI-4/5/6 pending) — the
>   crawler is "just another source of facts" feeding the same Validate → Ground → Human-review →
>   Enrichment pipeline.

## 1. The build, in plain language (what each piece is for)

| # | Piece | What it does for the owner |
|---|-------|----------------------------|
| **A** | **Manual strain entry** (KB page) | Staff type in a newly-discovered/ supplier-told strain with every field; it joins the verified strain list immediately — no deploy. |
| **B** | **Image-substitute KB** | A curated, owner-editable table of *approved* fallback images per category + product type, so a product card is **never blank**. Honest, compliant substitutes. |
| **C** | **Image source ladder + media pipeline** | For each product, try (1) exact image, (2) brand/vendor-specific unbranded-but-clear shot, (3) honest substitute, (4) category/type substitute from (B). All land as **draft** media on the product, employee approves. |
| **D** | **Vendors & brands enrichment** | Crawl/AI fill vendor info (about, region, contact, brands carried) and brand info (house style, signature lines, logo) as drafts on the Vendors/Brands pages. |
| **E** | **crawl4ai work-horse** | The engine: CSS-first extraction → `fit_markdown` → schema LLM extraction → verify-against-source → ground → compliance → map to POS key → draft in the existing review queue. Feeds B/C/D and strain enrichment. |
| **F** | **Review & measurement** | Everything crawler-produced shows in the same Bulk AI review grid; per-domain reliability + cost-per-accepted-field tracking. |

## 2. Slice plan (build order — one PR each)

> Ordered for *logical dependency* and *owner priority*: the two prerequisites first (manual strain
> entry, then image-substitute KB), then the media pipeline that consumes the substitute KB, then
> vendors/brands data, then the crawl4ai engine that feeds them all, then tuning. The AI-4/5/6
> hardening from the prior roadmap is woven in where it unblocks the crawler.

### **DF-1 — Manual strain entry on the KB page** ✅ *(prerequisite #1; build first)*
**Goal:** Add an "Add / edit a strain" form at the bottom of `/admin/knowledge-base` exposing every
field the `kb_strains` table uses: name, aliases, type (indica/sativa/hybrid/unknown), lineage,
aroma_notes, flavor_notes, terpenes, dominant_cannabinoid, potency_note, bud_structure, origin,
sources, confidence, summary, active. Slug derived from name (editable). NO brand field.
- New store fn `upsertKbStrain(input, actorId)` (upsert on `slug`) in `src/lib/ai/kb/store.ts`.
- New server action `upsertStrainAction` in `knowledge-base/actions.ts` (requirePermission
  `products.enrich`, audit, revalidate). Comma-split array fields; clamp confidence 0..1.
- UI: collapsible form section + make the existing Strains preview table rows **editable**
  (click-to-edit pre-fills the form) and add an active/disable toggle. Helper copy: "Sensory &
  factual only — no health/effect claims (WA rules)."
- Validation: required name + type; arrays trimmed/de-duped; friendly errors.
**Exit:** Staff can add a verified strain with all fields and see it appear in the list & counts;
re-adding the same slug updates it (idempotent). tsc + eslint + build green.
**No migration** (table already has all columns).

### **DF-2 — Image-substitute KB (data + admin)** *(prerequisite #2)*
**Goal:** Owner-editable approved fallback images per **category** and **inventory type**, so blank
cards get an honest substitute.
- **Migration `0021_image_substitutes.sql`** (idempotent): table `kb_image_substitutes`
  (`id`, `scope` = `category|inventory_type|global`, `key` text (e.g. `flower`,
  `concentrate for inhalation`, or `*`), `media_id` uuid → media table, `label`, `priority` int,
  `active` bool, audit cols). Unique on (`scope`,`key`,`media_id`). Partial index on active.
- Store helpers: `listImageSubstitutes`, `upsertImageSubstitute`, `setSubstituteActive`,
  `resolveSubstituteFor(category, inventoryType)` (category match → inventory-type match → global).
- Admin: new section/page "Fallback images" — upload a media asset, tag it to a category/type or
  global, set priority, preview the grid. Reuse existing media upload + `media` bucket.
- Seed the **full taxonomy** (all 52 categories + 6 inventory types) with sensible neutral
  placeholders so every product type is covered from day one (owner swaps in real art later).
**Exit:** For any (category, inventory type) we can resolve an approved substitute image; admin can
manage them. Migration handed to owner to apply.

### **DF-3 — Product image resolver + media draft pipeline**
**Goal:** Wire the **image source ladder** so a product card always has an image, with provenance.
- Resolver `resolveProductImage(posKey)` order: (1) product's own approved enrichment image →
  (2) brand/vendor approved generic image (branded-tube/jar/box shot) → (3) honest direct
  substitute → (4) `resolveSubstituteFor(category, type)` from DF-2. Returns `{url, source, isFallback}`.
- Front-end product card + PDP consume the resolver; show a subtle "representative image" note when
  `isFallback` (honest, never misleading).
- Crawler/AI-found images land as **draft** `image_media_ids` on `product_enrichment` (employee
  approves to publish), tagged `source = crawl:<url>` and a confidence.
- Brand/vendor "generic product shot" library: small table or reuse `kb_image_substitutes` with
  `scope=brand`/`scope=vendor` keyed by brand/vendor slug.
**Exit:** No product card is ever blank; fallback chain is provenance-tagged and honest.

### **DF-4 — Vendors & brands data model + admin enrichment**
**Goal:** Make vendor/brand pages ready to *receive* crawled/AI data as drafts.
- Audit existing vendors/brands tables; add (idempotent migration `0022_*` if needed) draft columns:
  about/description, region, website, logo_media_id, brands_carried[], `enrichment_status`
  draft/published, provenance/confidence.
- Admin Vendors/Brands: editable forms + "AI draft" badges + accept/reject, mirroring the product
  enrichment review UX. Brand house-style/known-for already partially in `kb_brands` — link them.
**Exit:** Vendor & brand records can hold AI/crawl drafts and be approved/published.

### **DF-5 — AI-4: provenance + confidence + accept-rate reporting** *(from prior roadmap)*
**Goal:** Make crawler ROI measurable before it scales.
- Surface `confidence`/`source` (already on `ai_suggestions`) in the review grid; add
  accept-rate-by-`prompt_version`/feature/source to `/admin/ai-usage`.
**Exit:** We can see which sources/prompts earn their keep (cost-per-*accepted*-field).

### **DF-6 — crawl4ai work-horse engine (foundation)**
**Goal:** Stand up the crawler that feeds B/C/D/strains. **LLM is last resort.**
- Python service (crawl4ai) OR a TS bridge calling a small crawl worker — decide per ops
  constraints; provider-agnostic via the same `AI_BASE_URL`/`AI_MODEL`.
- Pipeline: target+permission (robots, rate-limit, cache) → **CSS/XPath no-LLM extraction first** →
  `fit_markdown` pruning for messy pages → **schema LLM extraction** (Pydantic mirror of our Zod
  schemas, temp≈0, chunk+overlap) → **verify-against-source** (drop unsupported facts) → ground
  against cannabis KB → compliance scan → map to POS key → write `ai_suggestions` with
  `source=crawl:<url>` + confidence + (for images) candidate media.
- Targets v1: vendor/brand sites (owner-approved list), public product pages. Strain enrichment can
  reuse it to fill missing KB strains as drafts (still human-validated, no-guessing rule).
**Exit:** Crawled vendor/brand/image/strain drafts appear in the existing review queue; nothing
auto-publishes.

### **DF-7 — Crawl tuning + scale**
**Goal:** Keep it cheap, reliable, maintained.
- Per-domain reliability scoring (down-rank junk sources), scheduling/batch, de-dup, image candidate
  capture with vision alt-text, budget alerts.
**Exit:** High accept-rate, low cost, self-maintaining.

### **DF-8 (optional) — AI-5/6 hardening** *(from prior roadmap; do if crawler quality needs it)*
Golden-set eval harness + model router/budgets so prompt changes are gated by accept-rate.

## 3. Cross-cutting principles
- **Honesty on images:** substitutes must be obviously category-representative, never a fake of a
  specific product the customer would be misled by. Show a "representative image" cue on fallbacks.
- **Drafts only / human accept** everywhere. No auto-publish, ever.
- **No guessing on facts** (strains especially): only verified, cross-referenced data; blank beats wrong.
- **Idempotent migrations**, owner-applied. Keep a clear "APPLY THIS" note per migration.
- **Cost discipline:** cheapest path first (deterministic → CSS → small model → heavy), prune input,
  measure cost-per-accepted-field.
